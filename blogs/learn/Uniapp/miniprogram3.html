    初步了解微信小程序开发-基础篇3

    TuuOrange 5/3/2023 微信小程序uni-app

    # 自定义组件

    # 组件的创建

    1、 在项目的根目录中，鼠标右键，创建 components->test 文件夹

    2、 在文件夹上右键点击“新建 Component”

    3、 自动生成 4 个文件.js .json .wxml .wxss

    # 局部引用组件

    在页面的.json配置文件中引用组件的方式 - 局部引用

    "usingComponents": {
    "my-test1": "/components/test/test"
}

<my-test1></my-test1>

    # 全局引用组件

    在app.json中引用组件 - 全局引用

    "usingComponents": {
    "my-test1": "/components/test/test"
}

<my-test1></my-test1>

    # 组件与页面的区别

    • 组件的.json 文件中需要声明"component": true属性
    • 组件的.js 文件中调用的是Component()函数
    • 组件的事件处理函数需要定义到methods节点中

    # 自定义组件-样式

    # 组件样式隔离

    默认情况下，自定义组件的样式只对当前组件生效，不会影响到组件之外的 UI 结构：

    • 组件 A 的样式不会影响组件 B 的样式
    • 组件 A 的样式不会影响小程序页面的样式
    • 小程序页面的样式不会影响组件 A&B 的样式

    好处：

    1、 防止外界的样式影响到组件内部的样式

    2、 防止组件的样式破坏外界的样式

    # 组件样式隔离的注意点

    • app.wxss 中的全局样式对组件无效
    • 只有 class 选择器会有样式隔离效果，id 选择器、属性选择器、标签选择器不受样式隔离的影响

    建议：

    在组件和引用组件的页面中建议使用 class 选择器，不要使用 id、属性、标签选择器

    # 修改组件的样式隔离选项

    默认情况下，自定义组件的样式隔离特性能够防止组件内外样式互相干扰的问题。

    但是，有时候希望外界也可以控制组件内部的样式，需要通过styleIsolation修改组件的样式隔离选项

    // 1 在组件的 .js文件中添加设置
options: {
    styleIsolation: 'isolated'
},

// 2 在组件的 .json文件中添加设置
"styleIsolation": 'isolated'

    # styleIsolation 可选值

    可选值 默认值 描述
    isolated 是 表示启用样式隔离，在自定义组件内外，使用 class 指定的样式将不会互相影响
    apply-shared 否 表示页面 wxss 样式将影响到自定义组件，但自定义组件 wxss 中指定的样式不会影响页面
    shared 否 页面会影响自定义组件，自定义组件的样式会影响设置了 apply-shared 或者 share 的自定义组件和页面

    # 自定义组件-数据、方法、属性

    在小程序组件中，用于组件模块渲染的私有数据，需要定义到 data 节点中。

    在小程序组件中，事件处理函数和自定义方法需要定义到 methods 节点中。

    在小程序组件中，properties 是组件的对外属性，用来接收外界传递到组件中的数据。

    properties: {
    count: {
      type: Number,
      value: 0,
    },
    // count: Number, // 简化写法
},

<my-test count="1"></my-test>

    setData 也可以为 properties 的属性重新赋值。

    # 自定义组件-数据监听器

    数据监听器用于监听和响应任何属性和数据字段的变化，从而执行特定的操作。

    data: {
    count: 0,
    n1: 0,
    n2: 0,
    sum: 0
},
methods: {
  addN1() {
      this.setData({
        n1: this.data.n1 + 1
      })
    },
   addN2() {
      this.setData({
        n2: this.data.n2 + 2
      })
   }
},

// 属性和数据的监听器
observers: {
   // 可以监听多个字段的变化
   'n1, n2': function(n1, n2) {
     this.setData({
       sum: n1 + n2
     })
   }
},

    # 监听对象属性的变化

    数据监听器可以监听对象中单个或者多个属性的变化。

    为属性 A 或者为属性 B 或者为对象赋值都会触发监听器。

    // 监听对象中单个或多个属性的变化
'rgb.r rgb.g rgb.b': function(r, g, b) {
    this.setData({
      fullColor: `${r}, ${g}, ${b}`,
    });
}

    # 自定义组件-纯数据字段

    # 什么是纯数据字段

    指的是那些 不用界面渲染的 data 字段。

    应用场景：例如在有些情况下，某些 data 中的字段既不会展示在界面上，也不会传递给其他的组件，仅仅用于当前的组件内部，而这种 data 字段则为纯数据字段。

    # 使用规则

    options: {
   // 指定所有_开头的数据字段为纯数据字段
   pureDataPattern: /^_/
},
data: {
  a: true, // 普通数据字段
  _b: true, // 纯数据字段
}

    # 自定义组件-组件的生命周期

    生命周期函数 参数 描述说明
    created 无 在组件实例刚刚被创建时执行**（还不能调用 setData，在这个生命周期函数中，只应用于给组件的 this 添加一些自定义的属性字段）**
    attached 无 在组件实例进入页面节点树时执行，完全初始化完毕**（this.data 已经被初始化完成了，可以发送请求获取初始数据）**
    ready 无 在组件在视图层布局完成后执行
    moved 无 在组件实例被移动到节点树另一个位置时执行
    detached 无 在组件实例被从页面节点树移除时执行**（退出页面时，会触发页面中组件的 detached 生命周期函数，适合做一些清理工作）**
    error Object Error 每当组件方法抛出错误时执行

    # lifetimes 节点

    在小程序组件中，生命周期函数可以直接定义在 Component 构造器的第一级参数中，可以在lifetimes字段内进行声明（推荐用法，d 其优先级最高）。

    // 生命周期函数
lifetimes: {
   created() {
     console.log("created");
   },
   attached() {
     console.log("attached");
   },
   detached() {
     console.log("detached");
   },
},

    # 自定义组件-组件所在页面的生命周期

    # 什么是组件所在页面的生命周期

    有时候，自定义组件的行为依赖于页面状态的变化，所以需要用到组件所在页面的生命周期。

    比如：当触发页面的 show 生命周期函数时，组件中会有一些数据发生变化。

    在自定义组件中，组件所在页面的生命周期函数：

    生命周期函数 参数 描述
    show 无 组件所在页面展示时
    hide 无 组件所在页面被隐藏时
    resize Object Size 组件所在页面尺寸变化时

    # pageLifetimes 节点

    组件所在页面生命周期函数，需要定义在pageLifetimes节点中。

    // 组件所在页面生命周期函数
pageLifetimes: {
    show() {
      console.log("show");
      this.setData({
        'rgb.r': (Math.random() * 255).toFixed(0),
        'rgb.g': (Math.random() * 255).toFixed(0),
        'rgb.b': (Math.random() * 255).toFixed(0),
      })
    },
    hide() {
      console.log("hide");
    },
    resize() {
      console.log("resize");
    },
},

    # 自定义组件-插槽

    # 单个插槽 slot

    在小程序中，默认每个自定义组件只允许一个插槽站位 - 单个插槽。

    <test max="10">
  <!-- 放入的内容将放置在插槽slot的位置上 -->
  <view>页面插入slot</view>
</test>

<view class="wrapper">
  <view>组件内</view>
  <!-- 不确定的内容是用插槽slot进行占位，具体的内容由页面决定 -->
  <slot></slot>
</view>

    # 启用多个插槽

    在组件的.js 文件中配置进行启用。

    用不同的 name 区分不同的插槽。

    options: {
  multipleSlots: true, // 启用多个slot支持
}

<test max="10">
  <!-- 放入的内容将放置在插槽slot的位置上 -->
  <view slot="one">页面插入one slot</view>
  <view slot="two">页面插入 two slot</view>
</test>

<view class="wrapper">
  <view>组件内</view>
  <!-- 不确定的内容是用插槽slot进行占位，具体的内容由页面决定 -->
  <slot name="one"></slot>
  <slot name="two"></slot>
</view>

    # 自定义组件-父子组件之间的通信

    # 父子组件之间通信的 3 种方式

    1、 属性绑定

    • 用于父组件向子组件的指定属性设置数据，仅能设置 JSON 兼容的数据
    // 父
<view>父组件中的count：{{count}}</view>
<child count="{{count}}"></child>
// 子
<view>子组件中的count：{{count}}</view>
properties: {
   count: {
     type: Number,
     value: 100
   }
},

    2、 事件绑定

    • 用于子组件向父组件传递数据，可以传递任意数据
    // 父
<view>父组件中的count：{{count}}</view>
<!-- bind:自定义事件 传递给子组件 -->
<child count="{{count}}" bind:sync="syncCount"></child>

// 父组件中定义的传递给子组件的函数
syncCount(e) {
    // 同步count
    console.log("syncCount", e.detail.value);
   // 父组件中通过e.detail获取从子组件中传递过来的数据
   this.setData({
      count: e.detail.value,
    });
}

// 子
<view>子组件中的count：{{count}}</view>
<button bindtap="addCount">+1</button>

addCount() {
   this.setData({
      count: this.properties.count + 1,
   });
   // 在子组件的js中，调用this.triggerEvent('自定义事件', {/*参数对象*/})，将数据发送给父组件
   this.triggerEvent('sync', {value: this.properties.count};
}

    3、 获取组件实例

    • 父组件可以通过调用this.selectComponent("id/class选择器")获取子组件实例对象
    • 可以直接访问子组件中的任意数据和方法
    // 父
<view>父组件中的count：{{count}}</view>
<!-- bind:自定义事件传递给子组件 -->
<child count="{{count}}" bind:sync="syncCount" id="childRef"></child>
<button bindtap="getChild">获取子组件实例</button>

// 获取子组件实例
getChild() {
    const childRef = this.selectComponent("#childRef");
    childRef.setData({
      count: childRef.properties.count + 1,
    }); // 子组件的setData方法
    childRef.addCount(); // 子组件的addCount方法
}

    # 自定义组件-behaviors

    用于实现组件间代码共享的特性 === "vue - mixins"。

    # behaviors 的工作方式

    每个 behaviors 可以包含一组属性、数据、生命周期函数、方法。组件引用它时，会被合并到组件中。

    每个组件可以引用多个 behavior，behavior也可以引用其他的 behavior。

    # 创建 behavior

    调用Behavior(Object Object)方法即可创建一个共享的 behavior 实例对象，供所有的组件使用。

    // 调用behavior()方法，创建实例对象
// 并使用module.exports将behavior实例对象共享出去
module.exports = Behavior({
  // 属性节点
  properties: {},
  // 私有数据节点
  data: {},
  // 事件处理函数和自定义方法节点
  methods: {},
});

    # 导入并使用 behavior

    在组件中，使用require()导入需要的自定义 behavior 模块，挂载之后就可以访问 behavior 中的数据和方法。

    const myBehavior = require("../../behaviors/my-behavior");

Component({
  behaviors: [myBehavior],
});

    # 小程序对 npm 的支持与限制

    • 不支持依赖于 Node.js 内置库的包
    • 不支持依赖于浏览器内置对象的包
    • 不支持依赖于 C++插件的包

    # Vant Weapp

    小程序 UI 组件库，使用的是 MIT 开源许可协议，对商业使用比较友好。

    # 使用 npm 包-API Promise 化

    默认情况下，小程序官方提供的异步 API 都是基于回调函数实现的，容易造成回调地狱，影响代码的可读性和可维护性。

    通过额外的配置，将小程序的异步 API 升级为 Promise 的异步 API。

    依赖于第三方的miniprogram-api-promisenpm 包。

    // 在小程序入口文件app.js中调用一次 promisifyAll() 方法 就能够实现异步API的Promise化
import { promisifyAll } from "miniprogram-api-promise";

const wxp = (wx.p = {}); // 空白对象 指向同一份内存
promisifyAll(wx, wxp); // Promise化

    # 调用 Promise 化后的异步 API

    async getInfo() {
  const { data: res } = await wx.p.request({
    url: "",
    method: "",
    data: {}
  });
  console.log(data);
}

    # 全局数据共享

    全局数据共享是为了解决组件之间数据共享的问题。开发常用：vuex、redux、MobX....

    在小程序中，可以使用mobx-miniprogram（创建store）配合mobx-miniprogram-bindings（将store中的共享数据方法绑定到组件或者页面中使用）实现全局数据共享。

    // store.js
import { observable } from "mobx-miniprogram";

export const store = observable({
  // 数据字段
  count: 1,
  num: 0,
  // 计算属性
  get sum() {
    return this.count + this.num;
  },
  // actions方法，修改store中的数据
  updateNum: action(function (step) {
    this.num += step;
  }),
});

    # 将 store 中的成员绑定到页面中

    import { createStoreBindings } from "mobx-miniprogram-bindings";
import { store } from "../../store/store";

Page({
  onLoad: function () {
    // 生命周期函数 - 监听页面加载
    // this = 绑定到当前页面的实例上
    this.storeBindings = createStoreBindings(this, {
      store,
      fields: ["count", "num", "sum"],
      actions: ["updateNum"],
    });
  },
  onUnload: function () {
    // 生命周期函数 - 监听页面卸载
    this.storeBindings.destroyStoreBindings();
  },
});

    # 将 store 中的成员绑定到组件中

    import { storeBindingsBehavior } from 'mobx-miniprogram-bindings';
import { store } from '../../store/store';

Component({
  behaviors: [storeBindingsBehavior], // 通过storeBindingsBehavior实现自动绑定

  storeBindings: {
    store, // 指定绑定的store
    fields: {
      count: () => store.count, // 绑定字段的一种方式
      num: (store) => store.num, // 2
      sum: 'sum'， // 3
    },// 指定绑定的数据字段
    actions: {
      updateNum: 'updateNum'
    }, // 指定绑定的方法
  },
})

    # 分包

    分包是指将一个完整的小程序项目按照需求划分成不同的子包，在构建的时候打包成不同的包，用户在使用过的时候可以按需加载。

    分包后，小程序由一个主包+多个分包组成：

    • 主包：一般只包含项目的启动页面或者tabBar页面，以及一些公共资源
    • 分包：只包含和当前的分包有关的页面和私有资源

    # 分包的加载规则

    • 小程序启动时，默认下载主包和启动主包中的页面
    • 当用户进入分包某页面时，再进行对当前页面的下载

    # 分包的体积限制

    • 整个小程序的所有分包大小不能超过 16M（主包+分包）
    • 单个分包/主包不能超过 2M
    "pages": [
    "pages/list/list",
    "pages/home/home",
    "pages/index/index"

  ],
  "subPackages": [
    {
      "root": "pkgA",
      "name": "p1",
      "pages": [
        "pages/logs/logs"
    	]
    }
   ],

    分包的引用原则

    • 主包无法引用分包内的私有资源
    • 分包之间不能互享私有资源
    • 分包可以引用主包内的公共资源

    # 独立分包

    独立分包本质上也是分包，但是它比较特殊，可以独立于主包和其他分包而单独运行。（普通分包依赖于主包）

    # 独立分包的应用场景

    按照开发需求，可以将一些具有一定功能独立性的页面配置到独立分包中。（不依赖主包，可以提升分包页面的启动速度）

    一个小程序中可以有多个独立分包。

    # 独立分包的配置方法

    "subPackages": [
    {
      "root": "pkgA",
      "name": "p1",
      "pages": [
        "pages/dog/dog"
    	]
    },
  	{
      "root": "pkgB",
      "name": "p2",
      "pages": [
        "pages/apple/apple"
    	],
      "independent": true, // 通过independent，声明当前包为独立分包
    },
   ],

    # 分包预下载

    预下载分包的行为，会在进入指定的页面时触发。

    在 app.json 中，使用 preloadRule 节点定义为分包的预下载规则：

    {
  "preloadRule": {
    "pages/contact/contact": { // 触发分包预下载的页面路径
      "network": "all", // 默认wifi
        "packages": ["pkgA"], // 可以通过root或者name指定预下载的分包
    }
  }
}

    所有的分包预下载大小限额总和 2M。
