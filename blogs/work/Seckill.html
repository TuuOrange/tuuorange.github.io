    秒杀里我做了哪些操作

    TuuOrange 10/5/2024 工作工作总结学习学习总结秒杀

    # 秒杀前

    秒杀前要处理的问题：

    • 大量的用户刷新浏览页面
    • 页面倒计时
    • 访问流量到达高峰

    # CDN

    用 CDN 存放一切能提前存放的东西 - 但是我们部门没有支持

    # 擅用浏览器缓存

    • Nginx：Cache-Control: public, max-age=31536000, immutable
    • 文件哈希指纹：帮助精准缓存和更新
    • preload 预加载关键 css 之类的文件
    • 动态导入其他非关键功能，按需加载（懒加载），配合魔法注释
    • *service worker 存储静态资源
    • *indexDB 存储商品数据

    # 按需加载、懒加载非关键功能模块

    # 预先加载关键秒杀页面,css,js - preload/prefetch

    # 图片 CDN/OSS 或者懒加载，图片压缩、雪碧图（减少请求）、小图 base64 嵌入到代码中

    # indexDB 存储商品数据

    # 倒计时

    为了前端与服务端倒计时同步，需要在固定的时间内不停地向服务端确认倒计时时间。但是短轮询很影响性能

    同时，页面需要实时更新库存量，所以我创建了一个 websocket 连接，用来与服务器进行倒计时的匹配和更新以及库存量的更新。

    # 秒杀开始

    秒杀开始的时候需要处理的问题：

    • 防止 url 被获取，被盗刷
    • 请求过于频繁，导致服务器负荷过重

    # 防止 url 被获取盗刷

    # 签名 url

    sha256 加密 productId、userId、timestap...

    // 生成带签名的动态URL（Node.js示例）
const crypto = require("crypto");

function generateSeckillURL(productId, userId) {
  const timestamp = Date.now();
  const nonce = crypto.randomBytes(8).toString("hex");
  const params = `productId=${productId}&userId=${userId}&ts=${timestamp}&nonce=${nonce}`;

  // 使用HMAC-SHA256生成签名
  const hmac = crypto.createHmac("sha256", SECRET_KEY);
  hmac.update(params);
  const signature = hmac.digest("hex");

  return `/seckill/${productId}?${params}&sign=${signature}`;
}

// 输出示例：/seckill/1001?productId=1001&userId=123&ts=1620000000&nonce=a3d8e7f1&sign=9b8c7d6e...

    # token 化 url

    使用 JWT 生成 token 化 url

    // 使用JWT生成Token化URL
const jwt = require("jsonwebtoken");

function generateTokenURL(productId, userId) {
  const payload = {
    productId,
    userId,
    exp: Math.floor(Date.now() / 1000) + 30, // 30秒过期
  };
  const token = jwt.sign(payload, SECRET_KEY, { algorithm: "HS256" });
  return `/seckill/${token}`; // 示例：/seckill/eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
}

    # 节流

    节流是为了防止用户一直的点击发送请求，后面的请求覆盖前面的请求，同时给服务器造成更多压力

    点击请求之后过 2 秒才能继续请求

    可以将 setTimeout->requestAnimationFrame来实现

    # 设置请求队列管理

    当服务器压力多大时，前端可以设置请求队列处理并提示等待，避免瞬时洪峰。贯彻先进先出 FIFO 原则，在请求返回结果之后再执行下一次请求，防止前面的请求被覆盖，以及并发数太多给服务器太多压力（可以在队列中设置最大并发数）

    # 秒杀后

    秒杀后用户不停刷新订单或产生退单操作，大部分用户仍在刷新页面等待退单后的机会

    不过我们的项目不需要考虑到这一点，因为只是一个福利操作，兑换后不能够退换，是用积分领取的，一个人可以领取多个不同的商品

    # 总结

    静态资源尽量缓存，浏览器尽量缓存当前页面，如 html、css、js，图片都开启浏览器缓存

    1. CDN 分发静态资源存储到离用户最近的地方
    2. 文件名利用 hash 实现精确缓存（设置 filename 时添加 contenthash）
    3. 缓存头优化：Nginx：Cache-Control: public, max-age=31536000, immutable
    4. 资源预加载和预连接
    <!-- DNS预解析 -->
<link rel="dns-prefetch" href="//cdn.example.com">

<!-- 关键资源预加载 -->
<link rel="preload" href="/critical.css" as="style">
    1. 代码分割，按需加载，其他无关的页面进行按需加载和懒加载（动态引入非核心模块、懒加载）

    请求控制，避免崩溃

    1. 节流
    2. 设置请求队列和优先级控制

    实现同步实时更新，毫秒级更新

    1. 短轮询 - 不推荐
    2. sse - 服务端单方面推送
    3. websocket - 双向通信

    URL 动态化 - 为了解决接口防刷的问题 除了将 params 加密，也可以在 CDN 存放一个 js 脚本，然后前端发送请求执行这个脚本，这个脚本指向请求服务端实现真正的秒杀操作

    也可以，前端向服务端请求一个接口，服务端向前端返回一个加密后的接口地址（可以一次性），然后前端去请求这个地址实现真正的秒杀操作

    webworker - 复杂的计算放到 webworker 中计算，主线不卡顿
