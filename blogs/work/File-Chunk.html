    文件分片上传

    TuuOrange 3/11/2025 工作工作总结学习学习总结性能优化文件分片上传

    此处文件分片为基于转为 Blob/File 的图片进行的分片操作，但是也适用于大文件分片上传

    大文件分片上传核心其实是通过slice将二进制数据进行切分并发上传（并发上传需要控制），然后通过 hash 对上传的 chunk 进行标记以方便中断后继续上传和确认文件是否完全上传成功

    # 文件切片上传

    # 生成文件切片

    // 生成文件切片
createFileChunk(file, size = SIZE) {
  const fileChunkList = [];
  let cur = 0;
  while (cur < file.size) {
    fileChunkList.push({ file: file.slice(cur, cur + size) });
    cur += size;
  }
  return fileChunkList;
}

    # 设置 chunk 的 hash - 暂时用文件名+数组下标组成 hash

    async handleUpload() {
  if (!this.container.file) return;
    const fileChunkList = this.createFileChunk(this.container.file);
    this.data = fileChunkList.map(({ file }，index) => ({
      chunk: file,
      // 文件名 + 数组下标
      hash: this.container.file.name + "-" + index
    }));
  await this.uploadChunks();
}

    # 将切片上传至服务器

    // 上传切片
async uploadChunks() {
  const requestList = this.data
  .map(({ chunk，hash }) => {
    const formData = new FormData();
    formData.append("chunk", chunk);
    formData.append("hash", hash);
    formData.append("filename", this.container.file.name);
    return { formData };
  })
  .map(({ formData }) =>
    this.request({
    url: "http://localhost:3000",
    data: formData
    })
  );
  // 并发请求
  await Promise.all(requestList);
}

    Promise.allSettled 可以返回成功的和错误的请求，如果上传错误的话可以进行处理重新上传（可以限制重传次数 3 次）

    # 断点续传

    断点续传的原理在于前端/服务端需要记住已经上传了的切片，这样下次上传就可以跳过之前已上传的部分

    可以使用 localStorage 记录已经上传了的切片，但是换了浏览器就会失去记忆，不推荐

    # 服务器保存已经上传的切片的 hash

    这时候就不能够使用文件名+数组下标的组合格式了，因为文件名一旦更改就会失去效果。

    我们可以使用spark-md5来根据文件内容生成对应的 hash，只有文件内容更改才会改变 hash 值（参考了 webpack 中的指纹策略 contentHash）

    this.container.hash = await this.calculateHash(fileChunkList);
this.data = fileChunkList.map(({ file }，index) => ({
  fileHash: this.container.hash,
  chunk: file,
  hash: this.container.file.name + "-" + index,
  percentage: 0
}));

    # 文件秒传

    文件秒传的原理其实就是如果该文件已经上传成功了，那么就不需要继续上传了，直接返回已上传成功就行 - 这就是文件秒传

    const { shouldUpload } = await this.vertifyUpload(this.container.file.name, this.container.hash);
if (!shouldUpload) {
  this.$message.success("skip upload：file upload success");
  return;
} // 上传hash和文件名后端返回是否已上传成功

    # 暂停上传

    暂停上传就是清空队列中的 xhr，利用abort暂停所有的工作

    # 恢复上传

    恢复上传的原理就是在每次需要上传之前先vertifyUpload文件的上传情况，确定已经上传了的文件的 hash-chunk，然后过滤掉已上传的 hash-chunk（后端返回），继续上传未上传的 hash-chunk

    # 发送合并请求

    在队列已经都上传完成之后，前端主动向后端发送合并请求（带 file.name），后端对 chunk 进行合并

    # 显示上传进度条

    # 单个切片进度条

    createProgressHandler(item) {
  return e => {
    item.percentage = parseInt(String((e.loaded / e.total) * 100));
  };
}

    request.onProgress: this.createProgressHandler(this.data[index])

    如果该 chunk 切片原本已经上传成功了，则 perventage 初始值直接为 100，否则为 0

    # 总进度条

    总进度条就是将切片的进度（已上传的部分）进行累加 / 整个文件的大小（added / total）

    设置一个假的进度条：如果有已经上传过的切片，再重新上传，进度条会缩回去。所以要设置一个假的进度条，取假的进度条（数值大），假的进度条会保存整个的所有的进度

    参考自：比较完整的文件分片、断点、妙传方案https://juejin.cn/post/6844904046436843527?searchId=202504220100017C367FA810659694BB73

    # 补充 - 并发控制上传

    可以使用Promise.race先上传最快的，最继续进行上传，直到队列的数量可以在控制范围内（这是我看到的别人的方案 - 用 race 应该也是为了防止存在阻塞）

    我自己想的方案：可以设置队列（比如说我想一次上传四个 chunk），那么我队列可以先取四个 chunk（其他的 chunk 设为等待 pending 状态），然后在一个 chunk 完成之后，我再补充下一个等待中的 chunk 继续（同时如果有发生错误的或者超时就取消再重传，控制次数为 3 次，超过 3 次为此次上传失败） - 就是始终保持是四个 chunk 的上传状态（直到队列中没有在等待的 chunk）
